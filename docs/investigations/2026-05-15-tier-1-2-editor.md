## Executive summary

C-2 is the first production consumer of three load-bearing substrate vocabularies (chrome v2, page-background substrate, typography) and the four C-1 visual-authoring primitives (ScrubbableButton, TokenSwatchPicker, ChromePresetPicker, PropertyPanel). The substrate phase is complete; the consumer phase begins here.

The investigation surfaces three high-leverage decisions for James to lock before any C-2 build prompt drafts:

1. **Replacement vs. rewrite vs. extend of `FocusEditorPage` (1,716 LOC).** The current page exists, mounted in Studio at `/studio/focuses` and `/studio/<vertical>/focuses`, but predates the sub-arc A Tier 1+2 substrate. It still authors against the legacy `focus_compositions` shape (Tier 3 full-replace, no inherits-from-core reference). It must be rewritten — but the recommendation is **rewrite-with-salvage**, not greenfield. The hierarchical browser, the `InteractivePlacementCanvas` (678 LOC; carries a documented canon lock against @dnd-kit for 2D-grid drag), the tab pattern, and the Funeral Scheduling preview harness all survive intact. What's rewritten: the data layer (consumes inheritance resolver + cores + templates services instead of compositions), the inspector composition (PropertyPanel + 3 sections of C-1 primitives replacing the old per-prop control rail), the create flow (from-Core seeding instead of registry-component selection), and the inheritance surfacing (entirely new chrome). Net effect: ~700-900 LOC of existing FocusEditorPage stays mostly intact; ~800-1,000 LOC of new code replaces the rest.

2. **Single editor with tier toggle vs. two editors.** Locked in the prior investigation (2026-05-14 `focus-template-inheritance.md`, decision 3): "One Studio rail section ('Focus Templates'), tier toggle inside." C-2 inherits this; do not relitigate. The rail key (`focuses`) is already wired. What this investigation surfaces is the *editor-internal* tier toggle UX — segmented control at the top of the editor, no left-rail change.

3. **Substrate plumbing (Q7).** A real architectural fork. The C-1.1 fix surfaced that `platform_themes.resolve_theme` returns an empty `{}` for missing rows; C-1's demo handles this with a `FALLBACK_TOKENS` hardcoded mirror. C-2 must commit to one of two paths: (a) seed canonical tokens into `platform_themes` on bootstrap so the resolver becomes the single source, OR (b) keep `tokens.css` as read-side source and have `platform_themes` carry tenant-level overrides only, with editor merging at read time. Path (b) preserves the "Tokens.css is a mirror" canon better; path (a) simplifies editor logic. Recommended lean is (b), but the decision must be locked because two future arcs (Spaces substrate, Theme editor refinements) depend on it.

The other 7 question areas (canvas component reuse, three-section inspector composition order, create-from-Core flow, inheritance surfacing, test strategy, LOC budget, decomposition points) have clear leans grounded in existing canon and codebase reality; James reviews to confirm or override.

## Q1: Studio rail integration

### Options

A. **Single rail entry "Focuses" with tier toggle inside the editor body.** Currently shipped. The Studio rail at `frontend/src/bridgeable-admin/components/studio/StudioRail.tsx:63` declares `{ editor: "focuses", label: "Focuses", icon: FocusIcon }`. Tier toggle (Tier 1 cores / Tier 2 templates) lives inside the editor as a top-bar segmented control or browser-level filter.

B. **Two rail entries "Focus Cores" and "Focus Templates".** Studio rail gains a second entry. `studio-routes.ts:STUDIO_EDITOR_KEYS` extends from `focuses` to `focus-cores` + `focus-templates`. URLs become `/studio/focus-cores` and `/studio/<vertical>/focus-templates`.

C. **"Focus" parent grouping holding all three tiers + future in-place mode.** Rail entry is expandable; Tier 1, Tier 2, and (eventually sub-arc D) Tier 3 are children. Requires StudioRail to support nested entries — currently it's a flat list of `RailEntry` records.

### Trade-offs

| Option | Pros | Cons |
|---|---|---|
| A | Already wired; matches prior locked decision; lowest churn; rail stays flat | Editor-internal tier toggle UX has to be discoverable enough to feel intentional |
| B | Each tier gets its own URL surface; cleaner for deep-linking | Diverges from 2026-05-14 decision 3; doubles the rail real estate for a primitive operators will most often enter via the create-from-Core flow (Tier 2 work dwarfs Tier 1 work in expected frequency); StudioRail refactor; PLATFORM_ONLY_EDITORS / studio-routes update |
| C | Future-proofs for sub-arc D mode toggle inside the same group | StudioRail does not support nested entries today; ~150-200 LOC refactor of StudioRail itself, outside C-2 scope; Tier 3 in-place editor mounts in Live mode, not Edit mode — the parent-group abstraction breaks across mode boundaries anyway |

### Recommended lean

**Option A** — single "Focuses" rail entry, tier toggle inside.

### Decision

**Locked by prior canon** (DECISIONS.md 2026-05-14 / `focus-template-inheritance.md` Decision 3: "One Studio rail section ('Focus Templates'), tier toggle inside.") The rail is already wired; C-2 inherits.

Open sub-question for James: where does the tier toggle render inside the editor? Three options:

- (i) Segmented control in the editor's top bar, leftmost ("Tier 1 cores | Tier 2 templates"). Persistent context.
- (ii) Tab pair at the top of the hierarchical browser ("Cores" / "Templates" tabs above the list). Browser scope.
- (iii) Filter dropdown in the browser header. Less prominent.

**Recommended lean (i)** — top-bar segmented control. Tier is a load-bearing context attribute that affects what the canvas does, what the inspector shows, and what the create button creates; it deserves persistent placement, not buried in browser chrome.

## Q2: Tier 1 vs Tier 2 editor differences

### Tier 1 (Focus Core) scope

A core authors:
- Identity: `core_slug` (immutable once created), `display_name`, `description`, `registered_component_kind` + `registered_component_name` (which React component this core represents).
- Default placement geometry: `default_starting_column`, `default_column_span`, `default_row_index`, `min_column_span`, `max_column_span`.
- `canvas_config` JSONB (cosmetic canvas-level config — gap, padding, etc.).
- `chrome` JSONB (chrome v2 vocabulary — preset/elevation/corner_radius/backdrop_blur/background_token/border_token/padding_token).

What a Tier 1 core does **NOT** carry: substrate, typography, accessory placements. Substrate and typography are Focus-level concerns owned by Tier 2 (locked in B-4 + B-5). The core IS the single placement on its canvas — there's nothing else to author at Tier 1 besides chrome.

### Tier 2 (Focus Template) scope

A template authors:
- Identity: `template_slug`, `display_name`, `description`, `scope` ∈ {platform_default, vertical_default}, `vertical` (required iff vertical_default).
- Inheritance ref: `inherits_from_core_id` + `inherits_from_core_version`.
- Accessory composition: `rows` JSONB (rows-shape placements, R-3.0 shape), `canvas_config` JSONB.
- Chrome overrides: `chrome_overrides` JSONB (cascades on top of Tier 1's chrome).
- Substrate default: `substrate` JSONB (B-4 — five canonical presets).
- Typography default: `typography` JSONB (B-5 — four canonical presets).

The inherited core appears on the Tier 2 canvas as a fixed-but-visible placement, movable/resizable within `min_column_span` / `max_column_span` bounds (locked decision 2026-05-13 — "movable/resizable only" at Tier 3, by extension the core is the same shape at Tier 2 — though there's an open question on whether Tier 2 can resize the inherited core; see below).

### Editor surface

| Surface | Tier 1 | Tier 2 |
|---|---|---|
| Top bar | Tier toggle + breadcrumb + Save | Same shape + create-from-core flow lives here |
| Hierarchical browser (left) | Flat list of all active cores | Cores grouped by `registered_component_kind` (or focusType extension) as categories; templates as children |
| Canvas (center) | Single core placement on minimal canvas; chrome inspector targets it | Full accessory canvas with inherited-core placement + operator-authored accessories; reuses `InteractivePlacementCanvas` |
| Inspector (right) | Chrome section only (Tier 1 has no substrate / typography) | Three sections: Chrome (with "inherited from Tier 1" indicators) + Substrate + Typography (both Focus-scoped, no inheritance source) |
| Preview pane | Live core-card preview applying resolved chrome (mirrors C-1 demo shape) | Same preview applies cascaded chrome + substrate + typography |

### Same canvas component, different mode?

**Recommendation: same `InteractivePlacementCanvas` component with a `tier` prop that gates:**
- Tier 1: render only the core placement, lock add/delete, allow geometry adjustment within core's own bounds. Single-row layout.
- Tier 2: render full rows, inherited core as one placement, operator-authored accessories as the rest. Full add/delete/reorder.

Avoids forking a 678-LOC component. The `tier` prop branches in three or four places (placement palette enablement, delete-button visibility, add-row enablement). Estimate: ~80-120 LOC of branching inside the existing canvas.

### Tier 1 chrome inspector vs Tier 2 chrome inspector

The two are visually similar (same `PropertyPanel` shell + chrome `PropertySection`), but semantically differ:

- **Tier 1 chrome inspector** authors `focus_cores.chrome` — these are the *defaults* the cascade flows from. No "inherited from" indicators because Tier 1 is the root of the chrome cascade. All values render as authored (no italic-or-dimmed style).
- **Tier 2 chrome inspector** authors `focus_templates.chrome_overrides` — these are *overrides on top of Tier 1*. Every field shows an "inherited from Tier 1 core" indicator when the override is unset; setting the override styles the field as "explicitly authored at this tier." Resetting an override clears it back to inherited. The resolver's `sources.chrome_sources` provenance feed drives this UI.

### Open question for James

Does Tier 2 have authority to override Tier 1's `default_starting_column` / `default_column_span` for the inherited core placement, or are those Tier-1-only? Locked decision 1 (2026-05-13) says "core is immutable in identity at Tier 3; movable/resizable only" — the geometry-mutability question at Tier 2 is undeclared.

**Recommended lean**: Tier 2 CAN reposition the inherited core within row bounds (consistent with "movable/resizable" at Tier 3), but CANNOT change `min/max_column_span` (that's an identity attribute, Tier 1 only). The Tier 2 row carrying the core writes its own placement override into `rows[<core-row>].placements[<core-placement>].starting_column` + `column_span`; the resolver cascade handles the rest.

`PropertyPanel`, `PropertySection`, `PropertyRow` are the shell. `ScrubbableButton`, `TokenSwatchPicker`, `ChromePresetPicker` are the controls. The `ChromePrimitivesDemoPage.tsx:200-260` composition pattern is the architectural reference — replicate that shape across three sections.

### Section order

Two reasonable orderings:

A. **Chrome → Substrate → Typography** (matches B-3/B-4/B-5 ship order; consistent with C-1 demo focus on chrome-first).
B. **Substrate → Chrome → Typography** (most-to-least global — substrate is Focus-wide backdrop, chrome is per-placement frame, typography is text-treatment-only).

**Recommended lean: A** (Chrome first). Chrome is the most frequently-edited surface (every placement has chrome; not every Focus uses substrate). Operators land on the editor expecting to tweak the visible card chrome; substrate is atmospheric and edited less often.

### Section collapse defaults

All three sections start expanded. Operators can collapse independently per section. State is **component-local `useState`**, NOT persisted across sessions. (Persisting expansion state across editor sessions is a polish item, not C-2 scope.)

### Placement-selection vs Focus-scope state mux

Critical detail. When operator selects a placement on the canvas:
- The **Chrome** section's context becomes the selected placement's chrome (cascaded). Chrome is per-placement.
- The **Substrate** and **Typography** sections stay Focus-scope-wide. They don't change with placement selection.

When operator clicks empty canvas (deselect):
- All three sections show Focus-scope state (Substrate, Typography unchanged; Chrome shows the Focus's default chrome — the inherited Tier 1 chrome + Tier 2 overrides without any per-placement layer).

Inspector header needs a small chrome strip to communicate context:

```
┌─ Inspector ─────────────────────┐
│ Focus: Scheduling FH (Tier 2)   │  ← always shown
│ ─────────────────────────────── │
│ Selected: 📌 Today widget       │  ← only when placement selected
│ ─────────────────────────────── │
│ Chrome ▾                        │
```

### Inheritance lineage display per section

Each section header carries the cascade source:

- Chrome section header: "Chrome • cascading from: scheduling-kanban-core (Tier 1 v3)" when at Tier 2.
- Substrate section header: "Substrate • Focus-level (Tier 2 default)" when at Tier 2.
- Typography section header: similar to substrate.

Per-row "inherited from" indicators (Q6 covers this in more depth) are a small icon or dimmed label next to fields whose values come from the parent tier.

### Decision

**Recommended lean locked unless James objects:**
- Section order: Chrome → Substrate → Typography.
- All start expanded.
- Placement-selection swaps Chrome's context only; Substrate + Typography are Focus-scope.
- Inspector header surfaces tier + selected placement (when any) + lineage per section.

## Q6: Inheritance surfacing

### Goals

Operator must visually understand at all times:
- Which tier they're authoring.
- What this tier inherits from.
- Per-field, which value is explicit vs inherited.
- That clicking the inherited core lets them inspect (read-only) its values — but to *edit* the core they must switch tiers.

### Surfaces

#### Editor-level chrome (top bar)

```
┌─ [Tier 1 cores | Tier 2 templates ] ─────────────────────────┐
│ Tier 2: Scheduling — FH evening shift  (vertical_default — funeral_home)
│ ← inherits from: scheduling-kanban-core (Tier 1 v3)          │
└──────────────────────────────────────────────────────────────┘
```

The lineage line is the load-bearing affordance. Clicking the inherited-core link offers options: (a) jump to Tier 1 editor on that core, (b) view core in read-only side panel without leaving Tier 2. **Recommended lean: side panel.** Tier-switching mid-edit is destructive of unsaved state; the side panel lets operators reference Tier 1 without losing context.

#### Canvas-level inherited-core treatment

The inherited core placement renders distinctly from operator-authored accessory placements:
- Subtle dashed border (vs solid border on accessories).
- Small "Core" badge in the corner.
- Locked-delete affordance (trash icon disabled with tooltip "Cores cannot be deleted; switch to Tier 1 to edit the core").
- Drag handle still active (Tier 2 can reposition; locked Q2 sub-question pending James).

#### Inspector-level per-row inheritance indicators

Three visual treatments per `PropertyRow`:

- **Explicit at this tier**: full opacity, primary content color. Row value is what's stored on this tier's row.
- **Inherited from parent tier**: dimmed (text-content-muted), small "↑ inherited" caption beside the value. Hover/click reveals "Override at this tier" affordance that creates an explicit value.
- **Default from preset**: subtle marker (small dot indicator) showing the value comes from the canonical preset, not authored. Rare at Tier 2 unless preset is freshly applied without customization.

Backend resolver already returns `sources.chrome_sources`, `sources.substrate_sources`, `sources.typography_sources` — these dicts map field names to tier-of-origin. C-2's inspector reads these to set the per-row indicator state.

#### "Reset to inherited" affordance

Each explicitly-overridden row in the inspector gains a small reset button (↺ icon, hover-revealed). Clicking it clears the override on this tier; the resolver re-applies the parent tier's value. Implementation: POST or PUT to the template/core service with that specific override field set to `None` or removed from the override dict.

### Open question for James

Tier 1 version changes — when Tier 1 advances v3 → v4, what happens to Tier 2 templates inheriting from v3?

Two options, paralleling workflow templates' locked-to-fork semantics:
- (a) **Live cascade** (current substrate state): templates always inherit from the *current active version* of their core. v4 ripples to all templates immediately. Templates carry `inherits_from_core_version` stamped at create time, but the resolver currently uses live current-version lookup.
- (b) **Versioned cascade Option B**: templates pin to specific core version; ripple is opt-in via a "core has advanced — accept update?" notification.

Locked decision 2026-05-14 #2 says: "Live cascade for v1; versioned cascade Option B close behind. Schema must include `inherits_from_template_id` AND `inherits_from_template_version` from day one — Option B must be additive."

So C-2 ships (a) live cascade. The Option B notification UX is not C-2 scope. The schema field is already there; no migration debt.

### Decision

**Mostly locked.** Lineage chrome at top, distinct visual treatment for inherited core on canvas, per-row inheritance indicators in inspector, reset-to-inherited affordance per row, live cascade only (Option B notification deferred).

Open: side-panel-on-click vs jump-to-tier for inspecting the core. Recommended lean: side panel.

## Q7: Substrate plumbing decision

### The architectural fork

C-1's demo page handles missing theme tokens with `FALLBACK_TOKENS` (a hardcoded mirror of `tokens.css` light-mode values). C-1.1 doc surfaced this: `platform_themes.resolve_theme` returns `{}` when no row exists for the requested mode/vertical/tenant tuple.

This works in C-1 because the demo is internal/dev. C-2 is production; it must commit to a plumbing approach.

### Option (a) — Seed canonical tokens into `platform_themes`

On bootstrap (migration or seed script), every token in `tokens.css` is written to `platform_themes` as a `platform_default` row for mode=light AND mode=dark. `tokens.css` becomes a *bootstrap* source — read once at seed time, mirror of `platform_themes`'s authoritative state. Subsequent edits via the theme editor write to `platform_themes`; reads happen from `platform_themes` everywhere.

Pros:
- Single source of truth at runtime.
- C-2's editor's chrome/substrate/typography resolved-token previews are guaranteed correct (no fallback divergence risk).
- Theme editor's edits propagate naturally — no special-case for "this token isn't in DB yet."

Cons:
- **Inverts the "Tokens.css is a mirror" canon** (CLAUDE.md "Tokens.css is a mirror" section explicitly says `tokens.css` and `DESIGN_LANGUAGE.md` are synchronized; the implication is DL is source, tokens.css is the implementation mirror, NOT that platform_themes is source).
- Adds a migration that writes ~80 tokens × 2 modes = ~160 rows to `platform_themes`.
- Any drift between `tokens.css` and the seeded rows becomes a real data inconsistency, not a documentation inconsistency.
- C-2's scope balloons — the seed migration becomes a load-bearing dependency.

### Option (b) — `tokens.css` stays read-side source; `platform_themes` is override-only

`tokens.css` remains the canonical static source for token values at the platform tier. `platform_themes` rows carry ONLY overrides — at vertical_default scope or tenant_override scope. The C-2 editor:

1. Fetches `platform_themes.resolve` for the active scope (returns the merged override stack, possibly empty `{}`).
2. Composes against a frontend constant `BASE_TOKENS` (mirrors `tokens.css` light/dark values; identical to C-1's `FALLBACK_TOKENS`).
3. The effective token map is `{...BASE_TOKENS, ...resolved}`.

Pros:
- Preserves "Tokens.css is a mirror" canon — DESIGN_LANGUAGE.md / `tokens.css` stay synchronized; `platform_themes` is a separate layer of overrides on top.
- No migration; no seed data inconsistency risk.
- Matches how `platform_themes` is documented in CLAUDE.md "Theme Resolution" section: "Overrides store ONLY the values that differ from the parent scope, not the full token set."
- The frontend's `BASE_TOKENS` constant has a vitest test asserting it matches `tokens.css` at test-build time (this gate already exists for C-1's catalog-completeness test).

Cons:
- Editor logic carries the merge code (small, ~30 LOC).
- Theme editor's edits at vertical_default+ work normally; edits to platform_default would have to go to `tokens.css` itself — which is a code change, not a DB edit. This means **the theme editor cannot edit platform-default tokens via runtime UI**. That may be a deliberate constraint (theme editing happens at vertical_default+; platform-default tokens are DL-canon authored in commits) or a real limitation.

### Recommended lean

**Option (b)** — `tokens.css` as canonical platform-default source, `platform_themes` as override layer.

Reasoning:
- Preserves canon ("Tokens.css is a mirror").
- Matches CLAUDE.md Theme Resolution semantics ("overrides store ONLY the values that differ").
- C-1.1 already established `surface-frosted`'s addition via DL + tokens.css mirror — that pattern works.
- Theme-editor-cannot-edit-platform-defaults is the *correct* constraint: platform defaults are commitable code, not editable runtime state. If James wants runtime platform-default editing, that's a separate substrate arc.

### Decision

**Open question — James decides.** Two paths have real trade-offs. The default lean is (b), but committing to it constrains a future Theme editor refinement arc (can't edit platform-default tokens through UI without revisiting this).

If James locks (b), C-2 ships a `BASE_TOKENS` frontend constant + a vitest test asserting it matches `tokens.css`. No backend changes.

If James locks (a), C-2 gains a migration (~120 LOC) and the editor logic simplifies marginally — but the migration must ship before C-2's editor work begins.

## Q10: Natural decomposition points

### Decomposition candidates

Three natural seams emerge from §Q9:

1. **Substrate/data layer** ⇄ **inspector authoring** seam: the resolver/services adapter layer and the create-from-Core flow are independent of the canvas + three-section inspector. They could ship first as a "wire-up + create flow" sub-sub-arc.
2. **Tier 1 editor** ⇄ **Tier 2 editor** seam: Tier 1 is meaningfully simpler (single placement, chrome only). It could ship before Tier 2 as a smaller surface and serve as a proving ground for the inspector composition.
3. **Save/load + inheritance surfacing** ⇄ **canvas + inspector live editing** seam: the inheritance lineage chrome + per-row indicators + reset affordance could ship after the core editor surfaces work, as a polish sub-sub-arc.

### Proposed sub-sub-arcs

#### C-2.1 — Tier 1 cores editor + create-from-Core flow

**Scope.** Studio rail entry's tier toggle wiring. Hierarchical browser for Tier 1 with create flow ("+ New core" with registered-component picker). Tier 1 editor surface — canvas with single core placement, single-section chrome inspector (composes C-1 primitives directly per `ChromePrimitivesDemoPage` pattern), preview pane, save/load via `/cores` endpoints. The Tier 2 toggle position renders a placeholder "Tier 2 editor coming in C-2.2."

**LOC estimate.** ~1,200-1,800 (production + tests).

**Dependencies.** Substrate plumbing decision (Q7) must be locked. Studio rail entry already wired (no work needed there).

**Ships visibly.** Yes — operators can author Tier 1 chrome defaults end-to-end. Useful even before C-2.2 lands because the canonical scheduling-kanban-core's chrome can be tuned in production.

#### C-2.2 — Tier 2 templates editor + three-section inspector + canvas adaptation

**Scope.** Tier 2 toggle position now mounts a full editor. Three-section inspector (chrome + substrate + typography) composing C-1 primitives + two new preset pickers (SubstratePresetPicker, TypographyPresetPicker — both built on the ChromePresetPicker pattern). Canvas adapted with tier prop, inherited-core visual treatment, accessory placement palette. Create-from-Core flow for Tier 2 (modal with core picker, scope/vertical fields). Save/load via `/templates` endpoints. Live cascade preview wiring.

**LOC estimate.** ~2,500-3,500 (production + tests).

**Dependencies.** C-2.1 must ship first (the editor shell + tier toggle + base inspector composition).

**Ships visibly.** Yes — operators can author Tier 2 templates end-to-end. The full chrome + substrate + typography vocabulary becomes operator-accessible.

#### C-2.3 — Inheritance surfacing + reset-to-inherited + polish

**Scope.** Lineage chrome at top of editor with click-for-side-panel read-only core view. Per-row inheritance indicators driven by resolver's `sources.*_sources` provenance. Reset-to-inherited per-row ↺ affordance. Dirty-state guards. Optional: side-panel for read-only Tier 1 inspection from inside Tier 2.

**LOC estimate.** ~700-1,200 (production + tests).

**Dependencies.** C-2.2 must ship first (the resolver provenance is consumed by the per-row indicators; the side panel reads core values via the same services).

**Ships visibly.** Polish layer — visible quality-of-life affordances, not net-new functionality. Operators can author at both tiers before C-2.3 lands; C-2.3 makes the authoring less error-prone.

### Recommended dispatch sequence

C-2.1 → C-2.2 → C-2.3, in order. No interleaving with other arcs. Each sub-sub-arc ships independently and is operator-visible.

### Reasoning

- Each sub-sub-arc falls under the ~2,000-2,500 worst-case production LOC ceiling that DECISIONS.md 2026-05-13 PM locked via the Studio 1a-i scoping precedent.
- C-2.1 establishes the load-bearing patterns (inspector composition, save/load adapters, tier toggle, hierarchical browser tier-awareness) once. C-2.2 inherits them.
- C-2.3 is genuinely separable because the inheritance surfacing reads from a backend provenance feed (`sources.*_sources`) that already exists post-B-5; nothing in C-2.1/C-2.2 needs to change to enable C-2.3.
- The order maps to operator value delivery: Tier 1 first lets the canonical scheduling-kanban-core's chrome be tuned in production; Tier 2 second lets vertical-default templates author the full substrate vocabulary; Tier 3 polish last refines the authoring experience.

### Why not single-dispatch

LOC budget alone forces decomposition (Q9 worst case ~3,800-6,500). Beyond that:
- The three sub-sub-arcs have meaningfully different scope shapes (data + create flow / canvas + inspector / surfacing + polish). Bundling them complicates review.
- C-2.1 ships a real production capability — Tier 1 chrome editing — without waiting for the full surface to land. This matches the "ship small visible increments" canon used across Phase R-1.6 and the Aesthetic Arc.

### Why not more aggressive decomposition (4+ sub-sub-arcs)

A 4-sub-sub-arc split (e.g., separating create-from-Core into its own arc) was considered and rejected. The flows are interlocked enough that splitting them produces sub-sub-arcs too small to justify the dispatch overhead. The 3-sub-sub-arc shape proposed lands each between ~700 and ~3,500 LOC — wide enough to fit independent dispatch comfortably without inviting fragmentation.
